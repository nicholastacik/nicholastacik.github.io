## Goal

A reproducible, interactive narrative of the whole project ‚Äî scrape ‚Üí embed ‚Üí cluster ‚Üí
name ‚Üí findings ‚Äî that a reader (and the author) can learn from. Every claim is backed by
a code cell reading a **committed artifact**; nothing re-scrapes or re-embeds at render
time.

## Core principle: the post reads outputs, never runs the pipeline

The post executes at Quarto render time (including in CI), so it must use **main deps
only** (`pandas`, `pyarrow`, `matplotlib`, plus `plotly` added here) and read only the
committed artifacts. No `torch`/`umap`/`sklearn`/`openai` ‚Äî those live in offline groups
and never touch CI. `freeze: auto` caches the executed output (committed under `_freeze/`)
so CI re-renders a post only when its source changes.

## Architecture

- File: `posts/jeopardy_ds/index.qmd`, executed by Quarto's jupyter engine.
- Code as **inline cells** in the `.qmd` (Quarto-idiomatic, matches the existing cheatsheet
  post), reading `clues.parquet`, `category_clusters.parquet`, `cluster_summary.parquet`,
  `cluster_labels.csv`, `category_tokens.parquet` via `pd.read_parquet` / `pd.read_csv`
  with paths relative to the post directory.
- `plotly` added to `[project.dependencies]` (renders self-contained interactive HTML in
  Quarto). Static charts use `matplotlib` (already a main dep).

## Structure (7 sections)

1. **Intro** ‚Äî the goal: using data to find what to study to get better at Jeopardy.
2. **Scraping J-Archive** ‚Äî the fetch/parse/crawl pipeline, politeness, ~563k clues
   (1984‚Äì2026), the schema. *Viz:* overview ‚Äî clues over time, `game_type` breakdown,
   round split (static matplotlib).
3. **From categories to documents ("tokenizing")** ‚Äî the category-instance unit (name +
   seeded-shuffled clue‚Üíanswer pairs), the local `bge-small-en-v1.5` embedding, and the
   offline-compute‚Üícommitted-artifact pattern that keeps CI reproducible.
4. **Clustering into "types"** ‚Äî KMeans + UMAP, the *k*‚âà50 choice. *Viz:* **the interactive
   plotly UMAP scatter** (hero) + a static cluster-size ranking bar.
5. **Naming the 50 types** ‚Äî the three-part fingerprints (top category names, exemplars,
   c-TF-IDF terms) ‚Üí the names. *Viz:* the table of 50 named types with sizes.
6. **What the data reveals / what to study** ‚Äî findings: wordplay is the biggest
   meta-domain (~10‚Äì13%); the k-sweep splitting wordplay into drill-vs-memorize; the
   applicability ranking (which types are studyable); the most common entities per type.
   *Viz:* applicability bar (static) + a few top-entity tables.
7. **A study tool** ‚Äî a link to the research tool (C2), with a "built next" note until it
   exists.

Findings prose is sourced from `posts/jeopardy_ds/notes/build-log.md`.

## Visualizations

- **Overview (static matplotlib):** clues-per-year line/bar, `game_type` value-counts bar,
  round split ‚Äî computed from the full `clues.parquet`.
- **Hero ‚Äî interactive plotly UMAP scatter:** a **stratified sample of ~10,000 points**
  (proportional per cluster) from `category_clusters.parquet`, colored by `cluster_id`,
  hover text = cluster name (from `cluster_labels.csv`) + an example category + a top
  entity (from `category_tokens.parquet`). Sampling keeps the embedded HTML/JSON small
  while density stays visually faithful; **statistics (sizes, counts) use the full data**.
- **Cluster-size ranking (static bar):** cluster sizes from `cluster_summary.parquet`,
  labeled with names.
- **The 50 named types (table):** name, size, a couple of top entities.
- **Applicability ranking (static bar):** `n_qualifying_phrases` per type from
  `category_tokens.parquet`, labeled with names.
- **Top-entity tables:** for a handful of illustrative types, their top phrases.

## Reproducibility

- Reads only committed artifacts; no network, no heavy deps, deterministic.
- `freeze: auto` (from `posts/_metadata.yml`); the executed `_freeze/` output is committed
  so CI renders deterministically and fast.
- `draft: true` keeps it out of listings/nav/search until promoted.

## Testing / verification

Quarto prose posts are not unit-tested. The verification is that the post **renders
without error and all figures appear**, via `uv run quarto render posts/jeopardy_ds/`
locally against the committed artifacts, with the resulting `_freeze/` output committed.
A reviewer checks the rendered HTML: the plotly scatter is interactive, tables/bars
populate, prose claims match the artifacts.
